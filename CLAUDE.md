# 工作区规范

## 工作流程

### 1. 会话开始

```text
<认识工作区>
- 回顾MEMORY.md和`history/`，了解项目的近期进度
- 检查`plan/`是否有未完成项，是否仍然有效，不盲目多开任务
</认识工作区>
↓
和用户对齐本次会话的目标，判断任务类型（新功能、Bug修复、重构与优化、文档整顿）；对于用户明确提出不走固定流程的极简任务，通常跳过2-4步
```

### 2. 明确`wanto/`

```text
询问用户的原始想法（想要做什么？为什么产生了这样的想法？）
↓
<收集需求>
- 新功能：功能描述、具体用途、细节
- Bug修复：具体Bug表现、影响范围
- 重构与优化：重构方向和期望效果
- 文档整顿：期望的最终状态？
</收集需求>
↓
显式定义边界和排除项（防止波及面过广），为每个需求设置独立的验收标准（面向用户可感知）
```

### 3. 完善`design/`

```text
调查研究（寻找参照项目和技术方案，复现漏洞，探索工作区）
↓
根据任务类型选择分支：
<新功能&重构与优化>
- 定义数据结构和接口、对象、特性等（取决于所用语言）
- 建立数据流和函数栈的管道图
- 拆分模块和最小函数
- 设计异常处理与容错策略
- 设计日志和通知
</新功能&重构与优化>
<Bug修复>
- 同时检查架构设计、契约定义和具体实现三层，找到真实病灶
- 全局检索同类问题，确保斩草除根
- 提前分析副作用，防止引入回归错误
</Bug修复>
<文档整顿>
- 根据生命周期和消费者区分信息性质，判断归属文档（`.docs/`、`DOCUMENTS/`、README.md、MEMORY.md）
- 分析是否有文档需要拆分、合并、去重或补全
- 对文档内部结构进行梳理和优化，对各模块要点进行简述
</文档整顿>
↓
对比不同方案，显式声明决策和原因供追溯（防止后续信息丢失）
```

### 4. 制定`plan/`

```text
根据`design/`列出具体任务清单
↓
根据依赖关系和操作量对任务进行分组，划分出执行阶段（先后顺序和并行关系）
↓
寻找执行风险并做好备案；为每个任务设置独立的验收标准（面向代码可验证）
```

### 5. 执行

```text
新功能&Bug修复：遵循最小性原则逐文件编修，并持续对照`design/`进行检查
重构与优化：小步修改，逐步验证等效（重构）和提升（优化）
文档整顿：逐条对照代码/配置/脚本核实文档陈述，边写边验证
↓
<功能验证>
- 基础设施允许时，进行类型安全检查和编译时检查
- 检查改动副作用和上下游兼容性
- 进行单元测试（检查当前模块与函数的自身行为）
- 按需进行回归测试（检查副作用和行为兼容性）
- 按需进行集成测试（检查模块间协作、数据流端到端）
</功能验证>
↓
<质量优化>
- 清理死代码、重复逻辑、多余空行和空格，统一缩进、大小写和标点符号
- 检查命名（变量名、函数名、文件名等）、语法、格式、注释，确保内容完整自明、表述清晰准确
- 检查各文件之间的依赖与被依赖、引用与被引用关系，整理**耦合性简报**分析潜在问题和处理方案
- 检查文档和注释是否与实际代码一致，整理**同步性简报**供用户审阅
</质量优化>
↓
再次进行回归检测以确保质量优化未影响功能实现
```

### 6. 收尾

```text
<整理文档>
- `wanto/`、`plan/`和执行时细节归档至`history/`
- 执行时调整回写至`design/`
- 补全与同步各级README.md
- 整理稳定结论进DOCUMENTS
- 关键记忆写入MEMORY.md（须向用户寻求许可）
</整理文档>
↓
检查配置模板（.env.example、config.default.yml等）和忽略规则（.gitignore、.dockerignore等）是否需要更新
↓
更新CHANGELOG.md和版本号
↓
向用户确认是否要commit & push
```

## 文档分工

### `@/.docs/` — 开发时文档

- **`./wanto/`**: 收集用户的原始需求和想法、项目未来的发展方向
- **`./design/`**: 记录技术栈、架构选型、接口定义、数据结构、函数划分等设计细节
- **`./plan/`**: 列出任务清单和执行顺序，设置deadline、检查项和验收标准
- **`./history/`**: 归档已完成任务的执行轨迹、决策取舍和被推翻的尝试；只记录不随时间变质的事实，遗留与待办归`wanto/`
- **`./references/`**: 存储开发过程中仅供参考的外部项目与文档资料
- **`./trashcan/`**: 需要清理无用内容时，助手只能将其移动到此文件夹，用户自己决定后续处理方式

### `@/DOCUMENTS/` — 完成时文档

开发完成后才整理的公开文档——清晰、准确、标准化的最终材料，包括技术栈说明、架构设计、API文档、用户指南、部署说明等。

### 各级`README.md` — 工作区索引

在关键目录记录各模块功能的自述文档，须保持内容的准确性和时效性；探索工作区时可作为导航指引优先阅读。

### 外部`MEMORY.md` — 持久化记忆

在`~/.claude/projects/${project}/memory`维护的历史上下文，作为备忘录沉淀用户偏好、项目决策、跨会话进度，方便快速进入工作状态；**仅记录真正能够指导长期工作的关键信息**，简洁说明做了什么、得到了什么结果，避免小作文。

## 语法糖

### 占位符

- `${...}`: 变量占位，表示可替换为具体值的内容。
- `$(...)`: 指令占位，表示需要根据说明生成或填写的内容。

## 注释风格

**详尽性**：尽可能为每一行有业务语义代码添加注释，不止解释“是什么”，还要声明“为什么”
**自明性**：注释应独立自明，不依赖跨文件的上下文
**一致性**：除Markdown格式外，所有文件的注释风格应严格保持一致

### 文件头

使用多行注释对文件名及路径、目的简述、关联文件（对该文件改动敏感、有依赖或耦合关系的文件）路径进行记录；无多行注释的文件格式使用连续单行注释作为代替。

```rust
//! @/.../${文件名} - ${文件目的简述}
//! - ${关联文件的相对路径}/${关联文件名}
//! - ./.docs/history/workspace_init.md
```

### 模块/子模块

使用多行注释分隔不同模块，并根据模块内容设置标题；对于部分文件格式，应额外加入对称的结尾符来统一风格；
使用`===`将模块标题注释行长度统一在32~48字符之间；使用`---`将子模块标题注释行长度统一在15~25字符之间。

```html
<!-- ========== ${模块标题} ========== -->
```

```yaml
# ---- ${子模块标题} ---- #
```

### 行内末尾

优先使用单行注释在行内末尾对文件主体内容进行解释说明；无单行注释时使用多行注释。

```css
.文件主体内容 { } /* $(按需添加在行内末尾对文件内容进行说明) */
```

```lua
local example = '文件主体内容' -- $(按需添加在行内末尾对文件内容进行说明)
```
